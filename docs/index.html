O Projeto

RPG de console com 2 classes de herói e 3 inimigos — Java puro, sem frameworks.

O desafio de design: cada aspecto do jogo precisava ser extensível e desacoplado sem criar uma hierarquia de herança impossível de manter.

  • Guerreiro e Mago com comportamentos de ataque distintos
  • Equipamentos que modificam atributos sem proliferar subclasses
  • Ações de turno reversíveis — habilidade exclusiva do Mago
  • Log e barras de HP atualizados automaticamente a cada evento
⚔ Strategy

Comportamento de ataque por classe — injetado no herói na criação

🛡 Decorator

Equipamentos somam bônus por composição, sem herança

↩ Command

Ações do turno como objetos — com execute() e undo()

📡 Observer

Strategy

Comportamento de Ataque

Sem o padrão

Um if/else gigante no loop de batalha: "se guerreiro, aplica crítico; se mago, ignora defesa…" — frágil e impossível de estender.

Com Strategy

O Hero recebe uma AttackStrategy na criação. Trocar o comportamento é trocar o objeto — o BattleEngine não sabe qual estratégia está ativa.

  • WarriorAttack — dano físico com 5% de crítico; usa constantes nomeadas
  • MageAttack — uma linha: return attacker.getAtk(); ignora o target
  • A interface recebe os Character completos — a strategy acessa qualquer atributo
AttackStrategy.java · WarriorAttack.java · MageAttack.java
public interface AttackStrategy {
    // Recebe objetos completos — não só primitivos
    int execute(Character attacker, Character target);
}

public class WarriorAttack implements AttackStrategy {
    private static final double CRIT_CHANCE      = 0.05;
    private static final double CRIT_MULTIPLIER  = 2.0;

    public int execute(Character attacker, Character target) {
        int base = attacker.getAtk() - target.getDef();
        if (random.nextDouble() < CRIT_CHANCE)
            return (int)(base * CRIT_MULTIPLIER);
        return Math.max(0, base);
    }
}

public class MageAttack implements AttackStrategy {
    public int execute(Character attacker, Character target) {
        return attacker.getAtk(); // target ignorado — ignora defesa
    }
}
Decorator

Equipamentos como Wrappers

Sem o padrão

Uma subclasse por combinação: GuerreiroComEspada, GuerreiroComEspadaEEscudo… Explosão combinatória de classes.

Com Decorator

Cada equipamento envolve o personagem anterior e delega tudo, sobrescrevendo apenas o atributo que modifica. Qualquer combinação funciona sem nova classe.

  • SwordDecorator — +10 ATK  |  ShieldDecorator — +8 DEF
  • AmuletDecorator — +5 ATK, +30 HP máximo (para o Mago)
  • Guerreiro: ShieldDecorator(SwordDecorator(h)) — dois wrappers empilhados
CharacterDecorator.java · SwordDecorator.java · BattleEngine.java
public abstract class CharacterDecorator extends Character {
    protected final Character wrapped;

    public CharacterDecorator(Character wrapped) {
        // Inicializa o super com valores do wrapped...
        super(wrapped.getName(), wrapped.getHp(),
              wrapped.getMaxHp(), wrapped.getAtk(), wrapped.getDef());
        this.wrapped = wrapped;
    }
    // ...mas sobrescreve todos os getters para delegar ao wrapped
    @Override public int  getAtk()          { return wrapped.getAtk(); }
    @Override public void takeDamage(int h) { wrapped.takeDamage(h); }
}

public class SwordDecorator extends CharacterDecorator {
    @Override
    public int getAtk() { return wrapped.getAtk() + 10; }
}

// Em BattleEngine.createHero() — composição em camadas:
h = new Hero(name, 120, 120, 20, 15, new WarriorAttack(), ...);
equipped = new ShieldDecorator(new SwordDecorator(h));
Command

Ações de Turno com Undo

Sem o padrão

Para reverter um ataque o engine precisaria guardar estado manual de cada operação — frágil, acoplado e impossível de generalizar para novos tipos de ação.

Com Command

Cada ação é um objeto com execute() e undo(). O CommandHistory mantém uma pilha LIFO — reverter é só desempilhar e chamar undo().

  • Mago tem 1 uso de undo por inimigo como habilidade especial
  • AttackCommand recebe os observers no construtor e notifica em execute()
  • undo() restaura o HP exato e imprime a mensagem de reversão
TurnCommand.java · AttackCommand.java
public interface TurnCommand {
    void execute();
    void undo();
}

public class AttackCommand implements TurnCommand {
    private final Character attacker, target;
    private final List<BattleObserver> observers;
    private int damageDealt; // salvo para reverter

    @Override public void execute() {
        damageDealt = attacker.attack(target);
        target.takeDamage(damageDealt);
        // Command dispara os Observers diretamente
        BattleEvent event = new BattleEvent(attacker, target, damageDealt);
        for (BattleObserver o : observers) o.onEvent(event);
    }

    @Override public void undo() {
        target.heal(damageDealt); // reverte o dano exato
        System.out.println("✦ " + attacker.getName()
            + " reverte o tempo - o dano foi desfeito.");
    }
}
Observer

Eventos de Batalha

Sem o padrão

O BattleEngine chamaria explicitamente o logger e o display em cada ação — acoplamento direto. Adicionar um novo listener exigiria modificar o engine.

Com Observer

O AttackCommand recebe a lista de observers e dispara um BattleEvent após cada ação. Adicionar um novo listener é só implementar a interface e registrar em setup().

  • BattleLogger — registra histórico completo da partida
  • StatusDisplay — exibe barras de HP após cada evento
  • BattleEvent — classe com campos final; imutável por design
BattleObserver.java · BattleEvent.java · BattleEngine.java
public interface BattleObserver {
    void onEvent(BattleEvent battleEvent);
}

// Classe com campos final — imutável sem ser record
public class BattleEvent {
    private final Character attacker, target;
    private final int hitTaken;

    public BattleEvent(Character a, Character t, int hit) { ... }
    public int getHitTaken() { return hitTaken; }
}

// Em BattleEngine.setup() — registro dos observers:
observers.add(new BattleLogger());
observers.add(new StatusDisplay());

// A notificação ocorre em AttackCommand.execute():
BattleEvent event = new BattleEvent(attacker, target, damageDealt);
for (BattleObserver observer : observers)
    observer.onEvent(event); // broadcast para todos

Como Tudo se Conecta

BattleEngine.setup() │ ├─ createHero() │ ├─ new Hero(..., new WarriorAttack()) ◀ Strategy injetada │ └─ equipped = ShieldDecorator(SwordDecorator(h)) ◀ Decorator empilhado │ └─ observers.add(new BattleLogger()) ◀ Observer registrado observers.add(new StatusDisplay()) BattleEngine.loop() — a cada turno: │ └─ new AttackCommand(equipped, enemy, observers) ◀ Command criado │ ├─ execute() │ ├─ attacker.attack(target) → Strategy.execute() chamada aqui │ ├─ target.takeDamage() → Decorator delega ao wrapped │ └─ for observer → onEvent() → Observer notificado aqui │ └─ [Mago] undo() └─ target.heal(damageDealt) → HP revertido com precisão
