O Projeto

RPG de console com 2 classes de herói e 3 inimigos — Java puro, sem frameworks.

O desafio de design: cada aspecto do jogo precisava ser extensível e desacoplado sem criar uma hierarquia de herança impossível de manter.

  • Guerreiro e Mago com comportamentos de ataque distintos
  • Equipamentos que modificam atributos sem proliferar subclasses
  • Ações de turno reversíveis — habilidade exclusiva do Mago
  • Log e barras de HP atualizados automaticamente a cada evento
⚔ Strategy

Comportamento de Ataque

1 · 2
Sem o padrão

Um if/else gigante no loop de batalha: "se guerreiro, aplica crítico; se mago, ignora defesa…" — frágil e impossível de estender.

Com Strategy

O Hero recebe uma AttackStrategy na criação. Trocar o comportamento é trocar o objeto — o BattleEngine nunca sabe qual estratégia está ativa.

  • A interface recebe Character completos — a strategy acessa qualquer atributo
  • WarriorAttack — 2 chances independentes: crítico (7%) e penetração (10%)
  • MageAttack — uma linha: ignora o parâmetro target completamente
Hierarquia Strategy no IronTurn
// Interface — contrato único entre Hero e qualquer ataque
public interface AttackStrategy {
    int execute(Character attacker, Character target);
}

AttackStrategy (interface)
│  + execute(attacker, target) : int
│
├── WarriorAttack
│     CRIT_CHANCE   = 7%   — dano × 2.0
│     PIERCE_CHANCE = 10%  — ignora DEF
│
└── MageAttack
      dano = atk — sempre ignora DEF


// Hero delega sem saber qual strategy está ativa:
@Override
public int attack(Character target) {
    return strategy.execute(this, target); // polimorfismo puro
}
Strategy

Comportamento de Ataque

2 · 2
  • PIERCE_CHANCE e CRIT_CHANCE são constantes nomeadas — sem números mágicos no código
  • A verificação de penetração ocorre antes do crítico — as duas chances são independentes e cumulativas
  • MageAttack ignora o parâmetro target; o BattleEngine não sabe disso — nem precisa
  • Adicionar uma nova classe (Arqueiro, Druida…) é só criar uma nova AttackStrategy — zero alteração no engine
Injeção em BattleEngine.createHero()

new Hero(name, 120, 120, 20, 15, new WarriorAttack(), ...)
new Hero(name, 80, 80, 30, 5, new MageAttack(), ...)

WarriorAttack.java · MageAttack.java
public class WarriorAttack implements AttackStrategy {
    private static final double CRIT_CHANCE    = 0.07; // 7%
    private static final double PIERCE_CHANCE  = 0.10; // 10%
    private static final double CRIT_MULTIPLIER = 2.0;

    @Override
    public int execute(Character attacker, Character target) {
        int base = (random.nextDouble() < PIERCE_CHANCE)
                ? attacker.getAtk()                     // ignora DEF
                : Math.max(0, attacker.getAtk() - target.getDef());

        if (random.nextDouble() < CRIT_CHANCE) {
            System.out.println("  ⚔  GOLPE CRÍTICO!");
            return (int)(base * CRIT_MULTIPLIER);
        }
        return base;
    }
}

public class MageAttack implements AttackStrategy {
    public int execute(Character attacker, Character target) {
        return attacker.getAtk(); // target ignorado — ignora defesa
    }
}
Decorator

Equipamentos como Wrappers

1 · 2
Sem o padrão

Uma subclasse por combinação: GuerreiroComEspada, GuerreiroComEspadaEEscudo… Explosão combinatória de classes impossível de manter.

Com Decorator

Cada equipamento envolve o personagem e delega tudo, sobrescrevendo apenas o atributo que modifica. Qualquer combinação funciona sem nova classe.

  • Guerreiro: 2 wrappers empilhados — Shield(Sword(hero))
  • Mago: 1 wrapper — Amulet(hero) — +5 ATK e +30 HP máximo
  • O engine só enxerga Character — não sabe quantos wrappers existem
Cadeia de delegação — Guerreiro com Espada e Escudo
// Chamada: equipped.getAtk()
// Percorre a pilha de wrappers até o Hero raiz:

ShieldDecorator.getAtk()  // → delega para wrapped
    └── SwordDecorator.getAtk()  // → wrapped.getAtk() + 10
            └── Hero.getAtk()  // → 20 (base)

// Resultado final: 20 + 10 = 30 ATK


// takeDamage() percorre o mesmo caminho:
ShieldDecorator.takeDamage(h) → wrapped.takeDamage(h)
    └── SwordDecorator.takeDamage(h) → wrapped.takeDamage(h)
            └── Hero.takeDamage(h) // HP modificado aqui

// O HP real vive no Hero. Decorators só delegam.
Decorator

Equipamentos como Wrappers

2 · 2
  • CharacterDecorator estende Character — mantém o contrato de tipo; o engine nunca precisa fazer cast
  • O construtor copia os valores do wrapped, mas todos os getters delegam para ele
  • SwordDecorator sobrescreve apenas getAtk() — todo o resto passa para o wrapped sem alteração
  • showEquipment() no engine itera a cadeia com instanceof para exibir a lista de itens equipados
Composição em BattleEngine.createHero()

equipped = new ShieldDecorator(new SwordDecorator(h));
equipped = new AmuletDecorator(h);

CharacterDecorator.java · SwordDecorator.java
public abstract class CharacterDecorator extends Character {
    protected final Character wrapped;

    public CharacterDecorator(Character wrapped) {
        super(wrapped.getName(), wrapped.getHp(),
              wrapped.getMaxHp(), wrapped.getAtk(), wrapped.getDef());
        this.wrapped = wrapped;
    }
    // Todos os getters delegam ao wrapped — HP real fica no Hero
    @Override public int  getAtk()          { return wrapped.getAtk(); }
    @Override public void takeDamage(int h) { wrapped.takeDamage(h); }
    // ... demais getters também delegam
}

// SwordDecorator — sobrescreve APENAS getAtk()
public class SwordDecorator extends CharacterDecorator {
    public SwordDecorator(Character wrapped) { super(wrapped); }

    @Override
    public int getAtk() { return wrapped.getAtk() + 10; }
    // Shield e Amulet: mesma estrutura; cada um sobrescreve 1 getter
}

Ações de Turno com Undo

1 · 2
Sem o padrão

Para reverter um ataque o engine precisaria guardar estado manual de cada operação — frágil e impossível de generalizar para novos tipos de ação.

Com Command

Cada ação é um objeto com execute() e undo(). O CommandHistory mantém uma pilha LIFO — reverter é só desempilhar e chamar undo().

  • O menu dinâmico usa MenuEntry(label, action, takesTurn) — ações são lambdas encapsuladas
  • AttackCommand e GuardCommand — dois Commands distintos no mesmo engine
  • Mago tem 1 uso de undo por inimigo como habilidade especial
TurnCommand.java · CommandHistory.java · BattleEngine.java (trecho)
public interface TurnCommand {
    void execute();
    void undo();
}

// Implementações no IronTurn:
TurnCommand
├── AttackCommand  // ataque + notifica observers + undo()
└── GuardCommand   // DEF ×2 por 1 turno + revert()

// Menu dinâmico — Command como lambda em MenuEntry:
private record MenuEntry(String label, Runnable action, boolean takesTurn) {}

// buildMenu() monta opções de acordo com a classe do herói:
menu.add(new MenuEntry("Atacar",   () -> new AttackCommand(...), true));
menu.add(new MenuEntry("Defender", () -> new GuardCommand(...),  true));  // Guerreiro
menu.add(new MenuEntry("Reverter", () -> history.undo(),         false)); // Mago
Command

Ações de Turno com Undo

2 · 2
  • damageDealt salvo como campo — o undo() reverte o dano exato causado, não um valor estimado
  • AttackCommand.execute() é também o ponto de integração com o Observer — notifica após aplicar o dano
  • GuardCommand usa revert() em vez de undo() — ciclo de vida gerenciado pelo engine via activeGuard
  • "Reverter Turno" chama history.undo() duas vezes — desfaz o ataque do inimigo e do herói
Decisão de design

GuardCommand não entra na pilha de CommandHistory — tem ciclo de vida próprio, revertido automaticamente no fim do turno.

AttackCommand.java
public class AttackCommand implements TurnCommand {
    private final Character attacker, target;
    private final List<BattleObserver> observers;
    private int damageDealt; // salvo para undo exato

    @Override public void execute() {
        damageDealt = attacker.attack(target);    // → Strategy.execute()
        target.takeDamage(damageDealt);            // → Decorator delega ao Hero

        BattleEvent event = new BattleEvent(attacker, target, damageDealt);
        for (BattleObserver o : observers)
            o.onEvent(event);                       // → Observers notificados
    }

    @Override public void undo() {
        target.heal(damageDealt); // reverte o dano exato
        System.out.println("✦ " + attacker.getName()
            + " reverte o tempo - o dano foi desfeito.");
    }
}
Observer

Eventos de Batalha

1 · 2
Sem o padrão

O BattleEngine chamaria explicitamente o logger e o display em cada ação — acoplamento direto. Adicionar um listener novo exigiria modificar o engine.

Com Observer

O AttackCommand recebe a lista de observers e dispara um BattleEvent após cada ação. Adicionar um listener é só implementar a interface e registrar em setup().

  • BattleLogger — registra histórico completo da partida
  • StatusDisplay — exibe barras de HP coloridas após cada evento
  • BattleEvent.Type — enum ATTACK / GUARD; observers reagem diferente por tipo
Fluxo de notificação — quem fala com quem
// Setup — registro em BattleEngine.setup():
observers.add(new BattleLogger());
observers.add(new StatusDisplay());

// Sem Observer — acoplamento direto (evitado):
// engine → logger.log(...)              → acoplado
// engine → display.show(...)            → acoplado
// engine → [novo listener]              → MODIFICAR O ENGINE

// Com Observer — AttackCommand faz o broadcast:
AttackCommand.execute()
    └─ for (BattleObserver o : observers) o.onEvent(event)
            ├── BattleLogger.onEvent()    // registra histórico
            └── StatusDisplay.onEvent()  // exibe barra de HP

// Adicionar listener: zero alteração no engine:
observers.add(new SoundEffect()); // basta registrar
Observer

Eventos de Batalha

2 · 2
  • BattleEvent é imutável — todos os campos são final, sem setters; observers não podem corromper o evento
  • O enum Type foi adicionado para distinguir ATTACK de GUARD sem criar subclasses de evento
  • Construtor de conveniência usa Type.ATTACK como padrão — código existente não precisou mudar ao adicionar GUARD
  • Cada observer decide o que ler do evento — BattleLogger usa getType() para exibir mensagem distinta em guarda
Open/Closed na prática

Adicionar o GuardCommand exigiu apenas inserir Type.GUARD no enum. Os observers existentes já recebem o tipo — basta um switch no onEvent().

BattleObserver.java · BattleEvent.java
public interface BattleObserver {
    void onEvent(BattleEvent event);
}

public class BattleEvent {
    public enum Type { ATTACK, GUARD }

    private final Character attacker, target;
    private final int       hitTaken;
    private final Type      type;

    // Construtor de conveniência — Type.ATTACK por padrão
    public BattleEvent(Character a, Character t, int hit) {
        this(a, t, hit, Type.ATTACK);
    }
    public BattleEvent(Character a, Character t, int hit, Type type) {
        // Todos os campos são final — imutável por design
        this.attacker = a;   this.target  = t;
        this.hitTaken = hit; this.type    = type;
    }
    public Type getType()     { return type;     }
    public int  getHitTaken() { return hitTaken; }
}
