O Projeto

RPG de console com 2 classes de herói e 3 inimigos — Java puro, sem frameworks.

O desafio de design: cada aspecto do jogo precisava ser extensível e desacoplado sem criar uma hierarquia de herança impossível de manter.

  • Guerreiro e Mago com comportamentos de ataque distintos
  • Equipamentos que modificam atributos sem proliferar subclasses
  • Ações de turno reversíveis — habilidade exclusiva do Mago
  • Log e barras de HP atualizados automaticamente a cada evento
⚔ Strategy

Strategy

Comportamento de Ataque

Sem o padrão

Um if/else gigante no loop de batalha: "se guerreiro, aplica crítico; se mago, ignora defesa…" — frágil e impossível de estender.

Com Strategy

O Hero recebe uma AttackStrategy na criação. Trocar o comportamento é trocar o objeto — o BattleEngine não sabe qual estratégia está ativa.

  • WarriorAttack — dano físico com 5% de chance de crítico (2× dano)
  • MageAttack — dano fixo que ignora a defesa do inimigo
  • Adicionar uma nova classe? Só implementar a interface
AttackStrategy.java · WarriorAttack.java · Hero.java
// Contrato: toda estratégia deve saber calcular dano
public interface AttackStrategy {
    int execute(int atk, int def);
}

public class WarriorAttack implements AttackStrategy {
    public int execute(int atk, int def) {
        boolean crit = Math.random() < 0.05;
        int dmg = Math.max(1, atk - def);
        return crit ? dmg * 2 : dmg; // crítico: 2× dano
    }
}

public class Hero extends Character {
    private AttackStrategy strategy; // injetado no construtor

    public int attack(Character target) {
        // Hero não sabe o que é a strategy — só chama
        return strategy.execute(getAtk(), target.getDef());
    }
}
Decorator

Equipamentos como Wrappers

Sem o padrão

Uma subclasse por combinação: GuerreiroComEspada, GuerreiroComEspadaEEscudo… Explosão combinatória de classes.

Com Decorator

Cada equipamento envolve o personagem anterior e delega tudo, sobrescrevendo apenas o atributo que modifica. Qualquer combinação funciona sem nova classe.

  • SwordDecorator  — +10 ATK
  • ShieldDecorator — +8 DEF
  • AmuletDecorator — +5 ATK, +30 HP máximo
  • Guerreiro recebe espada + escudo; Mago recebe amuleto
CharacterDecorator.java · SwordDecorator.java · Main.java
public abstract class CharacterDecorator extends Character {
    protected Character wrapped;
    public CharacterDecorator(Character c) { wrapped = c; }

    // Delega tudo por padrão — subclasses sobrescrevem o que mudam
    public int getAtk() { return wrapped.getAtk(); }
    public int getDef() { return wrapped.getDef(); }
}

public class SwordDecorator extends CharacterDecorator {
    public int getAtk() { return wrapped.getAtk() + 10; }
}

// Em Main.java — empilhamento de wrappers:
Character hero = new Hero("Arion", HeroClass.WARRIOR);
hero = new SwordDecorator(hero);   // ATK + 10
hero = new ShieldDecorator(hero); // DEF + 8
// hero.getAtk() → base + 10  |  hero.getDef() → base + 8
Command

Ações de Turno com Undo

Sem o padrão

Para reverter um ataque o engine precisaria guardar estado manual de cada operação — frágil, acoplado e impossível de generalizar para novos tipos de ação.

Com Command

Cada ação é um objeto com execute() e undo(). O CommandHistory mantém uma pilha LIFO — reverter é só desempilhar e chamar undo().

  • Mago tem 1 uso de undo por inimigo como habilidade especial
  • AttackCommand.undo() restaura o HP exato que foi subtraído
  • O engine nunca precisa saber o que é reversível — o comando sabe
TurnCommand.java · AttackCommand.java · CommandHistory.java
public interface TurnCommand {
    void execute();
    void undo();
}

public class AttackCommand implements TurnCommand {
    private int damageDealt; // guardado para reverter

    public void execute() {
        damageDealt = attacker.attack(target);
        target.takeDamage(damageDealt);
        notify(new BattleEvent(...)); // dispara observers
    }

    public void undo() {
        target.heal(damageDealt); // reverte o dano exato
    }
}

public class CommandHistory {
    private Deque<TurnCommand> stack = new ArrayDeque<>();

    public void undo() {
        if (!stack.isEmpty()) stack.pop().undo(); // LIFO
    }
}
Observer

Eventos de Batalha

Sem o padrão

O BattleEngine chamaria explicitamente o logger e o display em cada ação — acoplamento direto. Adicionar um novo listener exigiria modificar o engine.

Com Observer

O engine dispara um BattleEvent imutável para todos os observers registrados. Adicionar um novo componente é só implementar a interface e chamar addObserver().

  • BattleLogger — registra histórico completo da partida
  • StatusDisplay — exibe barras de HP após cada evento
  • BattleEvent — objeto imutável: tipo, ator, alvo, valor
BattleObserver.java · BattleEvent.java · BattleEngine.java
public interface BattleObserver {
    void onEvent(BattleEvent event);
}

// Record imutável — thread-safe por design
public record BattleEvent(
    String type, String actor,
    String target, int value
) {}

public class BattleEngine {
    private List<BattleObserver> observers = new ArrayList<>();

    private void notify(BattleEvent e) {
        observers.forEach(o -> o.onEvent(e)); // broadcast
    }
}

// Em Main.java — registro dos observers:
engine.addObserver(new BattleLogger());
engine.addObserver(new StatusDisplay());
